# Rate/Cost 制御ガイド（v1.10）

> 目的: 収集・生成パイプラインのコストとレートを**事前に抑制**し、予期せぬ失敗（429 / 5xx / タイムアウト）や予算超過を回避する。

## 基本方針
- **キャッシュ優先**: 同一キーの再取得は避ける（成果物ハッシュ `content_hash` / `provenance.hash` を活用）。
- **段階的なバックオフ**: 一時的な 5xx / ネットワーク障害は指数バックオフ（上限 60s, ジッタあり）。
- **並列度の上限**: Node ワーカー並列数は安全側（例: 4–8）。I/O 待ちの多い Step でも急激な拡大はしない。
- **サーキットブレーカー**: 連続失敗率が閾値超過（例: 20%）で該当ソースを**スキップ**して stub にフォールバック。
- **観測**: KPI に「取得件数 / キャッシュヒット率 / 失敗率 / 平均遅延」を出力。

## 実装ガイド
- **キャッシュキー**: `provenance`（`source/provider/id/collected_at/hash`）をキーとし、欠落時は stub へフォールバック。
- **ローカルキャッシュ**: `build/` / `public/app/` の既存成果物を**使い回し**（冪等性確保）。
- **再試行**: `fetch*`系ユーティリティに `maxRetries` / `backoff(ms)` を追加（既存呼び出しは既定値=低め）。
- **環境変数**（例）:
  - `RATE_MAX_CONCURRENCY`（既定: 4）
  - `RATE_RETRY_MAX`（既定: 3）
  - `RATE_BACKOFF_BASE_MS`（既定: 500）

## 運用ガイド
- **保守的スタート**: まずは低並列・短時間で**スモーク**。異常時はすぐ切り戻す（PR単位ロールバック）。
- **予算管理**: 1日当たりの試行上限（例: 2,000 call）を Docs に明記。しきい値超過でその日の処理を中断。

## KPI（Step Summary 出力例）
- `rate.calls` / `rate.cache_hits` / `rate.failures` / `rate.retry_avg` / `rate.p99_ms`
- キャッシュヒット率（目標: ≥50%）/ 失敗率（目標: ≤5%）

> v1.10 では**方針と最低限のKPI**を整備。実装の厳格化（スロットリング・集中監視）は v1.11+ で検討。
